# Практическая работа №4. Подключаемые сервисы и отказ от состояния

## Тема проекта

Веб-приложение **«Каталог альбомов»**.

Приложение позволяет хранить музыкальные альбомы со следующими полями:

- название альбома;
- исполнитель;
- год выпуска;
- оценка.

## Цель практики

Научиться подключать внешние ресурсы как прикрепляемые сервисы через конфигурацию и сделать приложение stateless.  
Всё состояние приложения должно храниться во внешней базе данных, а не внутри контейнера приложения.

---

## Шаг 1. Добавление базы данных через Docker Compose

В проект добавлен файл `docker-compose.yml`.

В нём описаны сервисы:

| Сервис | Назначение |
|---|---|
| `app` | Flask-приложение «Каталог альбомов» |
| `db` | PostgreSQL-база данных |
| `nginx` | Reverse proxy для доступа к нескольким экземплярам приложения |

PostgreSQL доступен для приложения по имени сервиса `db`.

Все сервисы подключены к общей сети `album-network`.

Для хранения данных PostgreSQL используется volume `postgres_data`, поэтому данные сохраняются после перезапуска контейнеров приложения.

---

## Шаг 2. Подключение к БД через переменные окружения

В приложении добавлена поддержка двух режимов работы с базой данных:

1. SQLite — для локального запуска без Docker.
2. PostgreSQL — для запуска в Docker Compose.

Переменные окружения для PostgreSQL:

| Переменная | Назначение |
|---|---|
| `DB_ENGINE` | Тип базы данных: `sqlite` или `postgresql` |
| `DB_HOST` | Хост базы данных |
| `DB_PORT` | Порт базы данных |
| `DB_NAME` | Имя базы данных |
| `DB_USER` | Пользователь |
| `DB_PASSWORD` | Пароль |

Для локального режима используется:

```text
DB_ENGINE=sqlite
SQLITE_DB_PATH=albums_dev.db
```

Для Docker Compose используется:

```text
DB_ENGINE=postgresql
DB_HOST=db
DB_PORT=5432
DB_NAME=albums_db
DB_USER=albums_user
DB_PASSWORD=albums_password
```

---

## Шаг 3. Перенос состояния из памяти в БД

В приложении не используются глобальные списки, словари или счётчики для хранения альбомов.

Все данные альбомов хранятся в таблице `albums`.

При запуске в Docker Compose состояние хранится во внешнем сервисе PostgreSQL.

Контейнеры `app` являются stateless:

- контейнер приложения можно удалить;
- можно запустить несколько экземпляров приложения;
- данные не пропадают, потому что они находятся в контейнере `db` и volume `postgres_data`.

---

## Шаг 4. Симуляция сбоя процесса

Сборка и запуск нескольких экземпляров приложения:

```bash
docker compose up --build --scale app=3
```

После запуска приложение доступно по адресу:

```text
http://localhost:8080
```

Проверка списка контейнеров:

```bash
docker compose ps
```

Остановка одного экземпляра приложения:

```bash
docker stop <имя_одного_контейнера_app>
```

После остановки одного контейнера остальные экземпляры приложения продолжают работать. Данные не пропадают, потому что они хранятся в PostgreSQL.

Для проверки можно открыть:

```text
http://localhost:8080
```

или:

```text
http://localhost:8080/health
```

---

## Шаг 5. Проверка stateless-приложения

Остановка всех контейнеров приложения:

```bash
docker compose stop app
```

Повторный запуск:

```bash
docker compose up --scale app=3
```

Данные сохраняются в PostgreSQL-volume `postgres_data`.

Если нужно полностью удалить контейнеры, но оставить данные:

```bash
docker compose down
docker compose up --scale app=3
```

Если удалить volume, данные будут удалены:

```bash
docker compose down -v
```

Эту команду используют только для полной очистки проекта.

## Проверка локального запуска с SQLite

Локально без Docker приложение может использовать SQLite:

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
python run.py
```

В `.env` можно оставить:

```text
DB_ENGINE=sqlite
SQLITE_DB_PATH=albums_dev.db
```

## Вывод

В ходе практической работы приложение было переведено на работу с подключаемым сервисом PostgreSQL.  
Состояние приложения хранится во внешней базе данных, а контейнеры приложения не хранят данные внутри себя.  
Это позволяет запускать несколько экземпляров приложения и перезапускать контейнеры без потери данных.
