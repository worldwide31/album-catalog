# Практика 3. Полное отделение конфигурации от кода

## Тема проекта

Веб-приложение **«Каталог альбомов»**.

Приложение позволяет добавлять, просматривать и удалять музыкальные альбомы.  
Поля альбома:

- название альбома;
- исполнитель;
- год выпуска;
- оценка.

## Цель практики

Полностью отделить настройки приложения от кода и научиться передавать конфигурацию через переменные окружения, чтобы один и тот же Docker-образ мог работать в разных средах: `dev`, `staging`, `prod`.

---

## Шаг 1. Выявление «зашитых» значений

В приложении были выделены значения, которые могут отличаться в разных окружениях:

| Переменная | Назначение | Пример |
|---|---|---|
| `APP_NAME` | Название приложения в интерфейсе | `Каталог альбомов DEV` |
| `HOST` | Адрес, на котором Flask слушает запросы | `0.0.0.0` |
| `PORT` | Порт приложения внутри контейнера | `5000` |
| `FLASK_ENV` | Режим работы приложения | `development` или `production` |
| `SECRET_KEY` | Секретный ключ Flask | `example-secret-key` |
| `DATABASE_URL` | Строка подключения к базе данных | `sqlite:///albums.db` |
| `CONFIG_FILE` | Имя YAML-файла локальной конфигурации | `config.yaml` |

API-ключи и внешние сервисы в текущем приложении не используются.

---

## Шаг 2. Получение значений из переменных окружения

В файле `app/config.py` добавлена функция `get_setting()`.

Порядок приоритета:

1. переменная окружения;
2. значение из `config.yaml`;
3. значение по умолчанию.

Это позволяет запускать приложение локально без обязательного задания всех переменных.

---

## Шаг 3. Работа с секретами

Создан файл `.env.example` с примерами переменных.

Файлы с реальными секретами добавлены в `.gitignore`:

```text
.env
config.yaml
```

Это значит, что реальные значения секретов не должны попадать в GitHub.

---

## Шаг 4. Менеджер конфигурации

Для Python-проекта подключена библиотека `dynaconf`, которая выступает аналогом `spf13/viper`.

Она позволяет:

- использовать YAML-файл для локальной разработки;
- хранить значения по умолчанию;
- переопределять настройки через переменные окружения.

Переменные окружения имеют приоритет над YAML-файлом.

---

## Шаг 5. Передача переменных в контейнер

Сборка Docker-образа:

```bash
docker build -t album-catalog:config .
```

Запуск контейнера с переменными окружения:

```bash
docker run --rm --name album-catalog-dev -p 8080:5000 ^
  -e APP_NAME="Каталог альбомов DEV" ^
  -e PORT=5000 ^
  -e FLASK_ENV=development ^
  -e SECRET_KEY=dev-secret-example ^
  -e DATABASE_URL=sqlite:///albums_dev.db ^
  album-catalog:config
```

После запуска приложение доступно по адресу:

```text
http://localhost:8080
```

---

## Шаг 6. Проверка независимости

Один и тот же образ можно запустить с другими настройками без пересборки:

```bash
docker run --rm --name album-catalog-prod -p 8080:7000 ^
  -e APP_NAME="Каталог альбомов PROD" ^
  -e PORT=7000 ^
  -e FLASK_ENV=production ^
  -e SECRET_KEY=prod-secret-example ^
  -e DATABASE_URL=sqlite:///albums_prod.db ^
  album-catalog:config
```

Образ тот же самый: `album-catalog:config`.

Меняются только переменные окружения:

- другое название приложения;
- другой порт внутри контейнера;
- другой режим работы;
- другая база данных.

Пересборка образа не требуется.
