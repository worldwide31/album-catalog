# Практическая работа №5. Сборка, релиз, запуск и масштабирование

## Тема проекта

Веб-приложение **«Каталог альбомов»**.

Приложение позволяет добавлять, просматривать и удалять музыкальные альбомы.

## Цель практики

Освоить разделение этапов сборки и выполнения, а также обеспечить возможность горизонтального масштабирования простым добавлением процессов.

---

## Шаг 1. Автоматизация сборки с уникальной меткой

В проект добавлен workflow:

```text
.github/workflows/docker-image.yml
```

Пайплайн запускается при push в ветку `main`.

Он выполняет следующие действия:

1. забирает код из репозитория;
2. собирает Docker-образ;
3. присваивает образу уникальные теги:
   - `sha-<commit_hash>`;
   - `build-<run_number>`;
   - `latest`;
4. публикует образ в GitHub Container Registry.

Пример итогового образа:

```text
ghcr.io/worldwide31/album-catalog:sha-a1b2c3d
```

---

## Шаг 2. Разделение сборки и конфигурации

В `Dockerfile` нет переменных окружения, которые относятся к staging или production.

Docker-образ содержит только:

- код приложения;
- зависимости;
- команду запуска.

Все настройки передаются во время запуска контейнера:

- `APP_NAME`;
- `APP_VERSION`;
- `RELEASE_ENVIRONMENT`;
- `SECRET_KEY`;
- параметры PostgreSQL;
- параметры Redis.

Благодаря этому один и тот же образ можно использовать для разных окружений.

---

## Шаг 3. Релиз как комбинация образа и конфигурации

Для релиза добавлен файл:

```text
docker-compose.release.yml
```

Он не собирает образ заново, а использует готовый образ из registry:

```yaml
image: ${APP_IMAGE}
```

Также добавлен workflow:

```text
.github/workflows/release.yml
```

Он принимает:

- окружение;
- тег образа.

Релиз идентифицируется комбинацией:

```text
APP_IMAGE + RELEASE_ENVIRONMENT + APP_VERSION
```

Пример:

```text
Image: ghcr.io/worldwide31/album-catalog:sha-a1b2c3d
Environment: staging
Version: sha-a1b2c3d
```

## Шаг 4. Масштабирование через увеличение числа процессов

Для локальной проверки используется Docker Compose.

Запуск трёх экземпляров приложения:

```bash
docker compose up --build --scale app=3
```

Доступ к приложению выполняется через Nginx:

```text
http://localhost:8080
```

Проверка распределения запросов:

```bash
python scripts/load_test.py
```

Скрипт обращается к endpoint:

```text
http://localhost:8080/health
```

В ответе есть поле:

```text
instance
```

Если запросы попадают на разные контейнеры, в выводе будут разные значения `instance`.

## Шаг 5. Отсутствие «липких сессий»

В приложение добавлен endpoint:

```text
/session-demo
```

Он увеличивает счётчик в пользовательской сессии.

Сессии хранятся не в памяти процесса Flask, а в Redis:

```text
SESSION_TYPE=redis
SESSION_REDIS_URL=redis://redis:6379/0
```

Это значит, что запросы одного пользователя могут попадать на разные экземпляры приложения, но сессия не потеряется.

Redis описан в `docker-compose.yml` и `docker-compose.release.yml` как отдельный подключаемый сервис.

## Основные команды проверки

### Сборка и запуск

```bash
docker compose up --build
```

### Масштабирование

```bash
docker compose up --build --scale app=3
```

### Проверка экземпляров

```bash
python scripts/load_test.py
```

### Проверка сессий

Открыть несколько раз:

```text
http://localhost:8080/session-demo
```

Счётчик должен увеличиваться, даже если запросы обрабатываются разными контейнерами.

## Вывод

В ходе практической работы было настроено разделение сборки и запуска приложения.

Docker-образ собирается один раз и получает уникальный тег.  
Релиз выполняется как комбинация фиксированного образа и конфигурации окружения.  
Приложение можно масштабировать горизонтально, увеличивая количество контейнеров.  
Пользовательские сессии вынесены в Redis, поэтому приложение не требует sticky sessions.
